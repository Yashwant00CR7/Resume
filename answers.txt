Q1. Walk me through the end-to-end architecture of your Career Development
    Assistant. How does a resume PDF go from upload to final recommendation?

A1. • User uploads PDF via Streamlit → PyPDF extracts raw text
    • Text is chunked (recursive character splitter, ~500 tokens, 50 overlap)

Q26. Explain the difference between LoRA and full fine-tuning.

A26. • Full fine-tuning: updates ALL model parameters — requires full GPU memory
     • LoRA: freezes base model, injects small trainable low-rank matrices (A, B)
     • Math: W' = W + BA, where B ∈ R^(d×r), A ∈ R^(r×k), r << min(d, k)
     • Benefit: reduces trainable params by 90%+; trains on a single GPU
     • Choose LoRA when: limited compute, domain adaptation, avoiding catastrophic forgetting
     • Choose full FT when: massive dataset, need to change model behavior fundamentally

Q27. How do Transformer attention mechanisms work? Derive the computational
     complexity of self-attention and explain why it's quadratic.

A27. • Attention(Q, K, V) = softmax(QK^T / √d_k) × V
     • Q, K, V are linear projections of input X: Q = XW_Q, K = XW_K, V = XW_V
     • QK^T: matrix multiplication of (n × d) × (d × n) = n × n → O(n²d)
     • Softmax over n×n matrix → O(n²)
     • Final multiply (n×n) × (n×d) → O(n²d)
     • Total: O(n²d) — quadratic in sequence length n
     • Why quadratic: every token attends to every other token (all-pairs comparison)

---

Q28. What is the difference between cosine similarity and dot product
     similarity when comparing embeddings?

A28. • Cosine: cos(A, B) = (A·B) / (||A|| × ||B||) — normalized, range [-1, 1]
     • Dot product: A·B = Σ(a_i × b_i) — unnormalized, range (-∞, +∞)
     • Cosine ignores magnitude, measures only directional similarity
     • Dot product is affected by vector magnitude — longer vectors score higher
     • Choice matters when: embeddings are not L2-normalized (dot product biases toward longer docs)
     • Best practice: L2-normalize embeddings, then cosine ≡ dot product

---

Q29. Explain the bias-variance tradeoff.

A29. • Bias: error from oversimplified model assumptions (underfitting)
     • Variance: error from sensitivity to training data fluctuations (overfitting)
     • Total Error = Bias² + Variance + Irreducible Noise
     • Diagnosis: training loss high + val loss high → high bias (underfit)
     • Training loss low + val loss high → high variance (overfit)
     • Fix bias: more complex model, more features. Fix variance: regularization, more data

---

Q30. What is the vanishing gradient problem?

A30. • Gradients shrink exponentially through layers during backprop (∂L/∂w → 0)
     • Cause: repeated multiplication by values < 1 (sigmoid/tanh activations)
     • LSTMs fix it: gating mechanism (forget/input/output gates) controls gradient flow
     • Transformers fix it: (1) residual connections (skip connections) bypass layers
     • (2) Layer normalization stabilizes activations at each layer
     • (3) Self-attention provides direct gradient paths to all positions (no sequential chain)

---

Q31. Explain cross-entropy loss.

A31. • Formula: L = -Σ y_i × log(ŷ_i) where y = true label (one-hot), ŷ = predicted prob
     • Binary CE: L = -[y·log(ŷ) + (1-y)·log(1-ŷ)]
     • Why not MSE: CE penalizes confident wrong predictions heavily (log → -∞ as ŷ → 0)
     • MSE gradient vanishes near 0 and 1 (sigmoid saturation) — CE doesn't
     • Gradient: ∂L/∂z = ŷ - y (elegant, no saturation when combined with softmax)
     • Optimizes for calibrated probabilities, not just correct ranking

---

Q32. What is batch normalization?

A32. • Normalizes layer inputs: x̂ = (x - μ_B) / √(σ²_B + ε), then y = γx̂ + β
     • μ_B, σ²_B computed per mini-batch during training
     • Accelerates training by reducing internal covariate shift
     • Training: uses batch statistics (mean, variance of current batch)
     • Inference: uses running averages (exponential moving average from training)
     • Allows higher learning rates and acts as a mild regularizer

---

Q33. Explain L1 vs L2 regularization.

A33. • L1 (Lasso): adds λΣ|w_i| to loss → gradient is ±λ (constant push toward 0)
     • L2 (Ridge): adds λΣw_i² to loss → gradient is 2λw_i (proportional push)
     • L1 sparsity: constant gradient pushes small weights exactly to 0 → feature selection
     • L2 shrinks all weights but never to exactly 0 → keeps all features, just smaller
     • Use L1 for feature selection / interpretability; L2 when all features are relevant
     • Elastic Net: combines both → λ₁|w| + λ₂w²

Q34. What are Sentence Transformers?

A34. • Fine-tuned BERT/RoBERTa models that produce fixed-size sentence embeddings
     • Vanilla BERT: outputs per-token embeddings → need pooling, not optimized for similarity
     • Sentence Transformers: trained with siamese/triplet networks on NLI + STS data
     • Output: single vector per sentence optimized for cosine similarity comparison
     • Training loss: contrastive loss or multiple negatives ranking loss
     • Key models: all-MiniLM-L6-v2 (fast), all-mpnet-base-v2 (quality), e5-large (SOTA)

Q35. Explain contrastive learning in embedding models.

A35. • Goal: pull similar pairs together, push dissimilar pairs apart in embedding space
     • Contrastive loss: L = (1-y)½D² + y½max(0, margin-D)² where D = distance
     • Triplet loss: L = max(0, D(anchor,positive) - D(anchor,negative) + margin)
     • InfoNCE (used in CLIP): L = -log(exp(sim(a,p)/τ) / Σexp(sim(a,n_i)/τ))
     • Multiple Negatives Ranking Loss: most efficient for Sentence Transformers
     • Key: quality of negatives matters — hard negatives improve embedding quality

Q36. What is knowledge distillation?

A36. • Training a small "student" model to mimic a large "teacher" model's behavior
     • Student trained on soft labels (teacher's probability distribution, not hard labels)
     • Loss: α × CE(student, teacher_soft) + (1-α) × CE(student, hard_labels)
     • Temperature τ > 1 softens teacher outputs → reveals inter-class relationships
     • For on-device: distill GPT-4 → DistilBERT-sized model for specific task
     • Benefits: 4-10x smaller model, 2-5x faster inference, minimal quality loss

Q37. Explain precision, recall, F1, and AUC-ROC.

A37. • Precision: TP / (TP + FP) — "of predicted positives, how many are correct?"
     • Recall: TP / (TP + FN) — "of actual positives, how many did we catch?"
     • F1: 2 × (P × R) / (P + R) — harmonic mean, balances both
     • AUC-ROC: area under TPR vs FPR curve — threshold-independent ranking quality
     • Optimize recall: medical diagnosis (don't miss cancer), fraud detection
     • Optimize precision: spam filtering (don't block legitimate emails)

---

Q38. How does gradient descent differ from Adam optimizer?

A38. • SGD: w = w - η × ∂L/∂w (fixed learning rate, no memory)
     • Adam combines: (1) Momentum — exponential moving avg of gradients (m_t)
     • (2) RMSProp — exponential moving avg of squared gradients (v_t)
     • Update: w = w - η × m̂_t / (√v̂_t + ε) where m̂, v̂ are bias-corrected
     • Adam: adaptive per-parameter learning rates, faster convergence
     • Trade-off: Adam may generalize worse than well-tuned SGD with scheduling

Q39. Generative vs discriminative models?

A39. • Discriminative: models P(y|x) directly — learns decision boundary
     • Examples: logistic regression, SVM, BERT classifier, most neural classifiers
     • Generative: models P(x, y) = P(x|y) × P(y) — learns data distribution
     • Examples: Naive Bayes, GANs, VAEs, GPT, diffusion models
     • Discriminative preferred: when you have lots of labeled data, care only about classification
     • Generative preferred: data augmentation, anomaly detection, creative generation

Q40. How does Mixture of Experts (MoE) work?

A40. • Architecture: multiple "expert" sub-networks + a gating/router network
     • Router: assigns each token to top-k experts (typically k=2 out of 8-64 experts)
     • Only selected experts compute for each token → sparse activation
     • Benefit: total model params can be huge (1T+), but per-token compute stays small
     • Mixtral-8x7B: 8 experts of 7B each, but only 2 active = 14B active params
     • Challenge: load balancing (preventing "expert collapse" — all tokens → 1 expert)

Q41. What is KL Divergence?

A41. • DKL(P || Q) = Σ P(x) × log(P(x) / Q(x)) — measures how Q diverges from P
     • Always ≥ 0; equals 0 only when P ≡ Q
     • Asymmetric: DKL(P||Q) ≠ DKL(Q||P)
     • VAEs: ELBO loss = Reconstruction Loss + DKL(q(z|x) || p(z)) — regularizes latent space
     • RLHF: KL penalty between fine-tuned policy and base model → prevents reward hacking
     • Intuition: penalizes the model for drifting too far from a reference distribution

---

Q42. Beam search vs greedy vs nucleus (top-p) sampling?

A42. • Greedy: always pick highest-probability token → fast, repetitive, deterministic
     • Beam search: track top-B sequences at each step → better global optimum, still repetitive
     • Top-p (nucleus): sample from smallest set whose cumulative prob ≥ p (e.g., p=0.9)
     • Top-p: more diverse, creative outputs — best for conversation/generation
     • Beam search: best for translation, summarization (where faithfulness matters)
     • Temperature: < 1 sharpens distribution (more deterministic), > 1 flattens (more random)

---

Q43. Why multiple attention heads?

A43. • Each head operates on a subspace: d_head = d_model / n_heads
     • Different heads learn different relationship types (syntactic, semantic, positional)
     • Single large attention: can only capture one weighted mix of relationships
     • Multi-head: parallel attention patterns → richer representation
     • Concatenated outputs are linearly projected: MultiHead = Concat(head_1,...,head_h) × W_O
     • Empirically: removing heads degrades performance; some heads are more interpretable

---

Q44. Pre-training vs fine-tuning vs in-context learning?

A44. • Pre-training: train on massive unlabeled data (next-token prediction) — learns language
     • Fine-tuning: update pre-trained weights on task-specific labeled data — adapts knowledge
     • ICL: provide examples in the prompt at inference — no weight updates
     • Pre-training: expensive ($M+), done once by labs (e.g., GPT-4, Llama)
     • Fine-tuning: moderate cost, best when you have 1K-100K labeled examples
     • ICL: zero cost, works with few-shot examples, but limited by context window

Q45. How do you evaluate LLM output quality beyond accuracy?

A45. • Ragas: measures faithfulness, answer relevancy, context precision for RAG
     • DeepEval: G-Eval (LLM-as-judge), hallucination score, toxicity detection
     • BLEU/ROUGE: n-gram overlap with reference — good for translation/summarization
     • Human evaluation: Likert scales for helpfulness, coherence, harmlessness
     • Perplexity: measures how well model predicts held-out text (lower = better)
     • Custom evals: task-specific metrics (e.g., dependency resolution success rate)

A46. • Data lake: raw, unstructured/semi-structured data (S3, GCS) — schema-on-read
     • Data warehouse: structured, cleaned, transformed data — schema-on-write
     • Lake: cheap storage, flexible, good for ML (raw features, embeddings, logs)
     • Warehouse: fast queries, BI dashboards, SQL-optimized (BigQuery, Snowflake)
     • ML pipelines: lake for raw data ingestion → warehouse for curated feature tables
     • Modern: lakehouse architecture (Delta Lake, Iceberg) combines both

Q47. Batch vs stream processing?

A47. • Batch: process large volumes on a schedule (hourly/daily) — Spark, Airflow
     • Stream: process events in real-time as they arrive — Kafka, Flink, Spark Streaming
     • Lambda architecture: batch layer + speed layer (process in both, merge results)
     • Kappa architecture: everything is a stream (simpler, Kafka-based)
     • Design: use Kafka for ingestion → Flink for real-time + Spark for batch backfills
     • Trade-off: batch = simpler, cheaper; stream = lower latency, higher complexity

Q48. How does vector indexing (HNSW, IVF) work internally?

A48. • HNSW: Hierarchical Navigable Small World graph — multi-layer proximity graph
     • Search: start from top layer, greedily descend to nearest neighbor, refine at bottom
     • IVF (Inverted File): partition vectors into clusters via k-means, search nearest cluster(s)
     • HNSW: O(log n) search, high recall, high memory (graph in RAM)
     • IVF: lower memory, but recall depends on nprobe (number of clusters searched)
     • Trade-off: HNSW = better recall for <10M vectors; IVF-PQ = better for 100M+ (with compression)

Q49. CAP theorem?

A49. • Consistency: every read returns the most recent write
     • Availability: every request receives a response (even if not the latest)
     • Partition tolerance: system continues despite network failures between nodes
     • Theorem: can only guarantee 2 of 3 in a distributed system
     • Pinecone: prioritizes AP (available + partition-tolerant) — eventual consistency
     • Production ML: usually accept eventual consistency for lower latency

Q50. What is data skew and how to mitigate it?

A50. • Data skew: uneven distribution of data across partitions — one partition is 100x larger
     • Effect: one Spark task takes 10 minutes, others finish in 10 seconds → bottleneck
     • Mitigation: (1) salting — add random prefix to skewed keys to distribute them
     • (2) Adaptive Query Execution (AQE) in Spark 3.x — auto-coalesces partitions
     • (3) Broadcast joins for small tables instead of shuffle joins
     • (4) Custom partitioning: hash on a more uniform key

Q52. What is schema evolution?

A52. • Schema evolution: changing data schema (add/remove/rename fields) without breaking pipelines
     • Critical because: downstream consumers (ML models, dashboards) break on unexpected schemas
     • Forward compatibility: old code reads new schema (new fields ignored)

Q53. OLTP vs OLAP?

A53. • OLTP: Online Transaction Processing — short reads/writes, row-oriented (MySQL, PostgreSQL)
     • OLAP: Online Analytical Processing — complex aggregations, column-oriented (BigQuery, Redshift)
     • OLTP optimized for: INSERT/UPDATE single rows, high concurrency, low latency
     • OLAP optimized for: SELECT with GROUP BY, SUM, COUNT over millions of rows
     • Company role dataset: OLAP — read-heavy, aggregated queries, not real-time writes
     • In practice: OLTP for app backend → replicate to OLAP for analytics

Q54. ACID properties?

A54. • Atomicity: transaction is all-or-nothing (commit or rollback entirely)
     • Consistency: transaction moves DB from one valid state to another
     • Isolation: concurrent transactions don't interfere (serializable, repeatable read, etc.)
     • Durability: committed data survives crashes (written to disk/WAL)
     • NoSQL (MongoDB, DynamoDB): often sacrifice consistency for availability (BASE model)
     • Vector DBs (Pinecone): eventual consistency, no traditional ACID guarantees

Q55. How to implement CDC for real-time Pinecone updates?

Q56. Partitioning vs sharding?

Q58. Handling duplicates in streaming pipelines?

A58. • Exactly-once: each record processed precisely once (hardest guarantee)
     • At-least-once: records may be processed multiple times (easier, need dedup)
     • At-most-once: records may be dropped (simplest, usually unacceptable)
     • Dedup strategies: (1) idempotent writes (upsert by unique key)
     • (2) Bloom filter for probabilistic dedup (fast, some false positives)
     • (3) Kafka exactly-once: enable.idempotence=true + transactional producers

Q61. Row-oriented vs columnar storage?

Q63. Idempotency in data pipelines?

A63. • Idempotent: running the same job twice produces the same result (no duplicates)

     • (4) Track processed offsets (Kafka consumer offsets, checkpoint files)

Q79. Python GIL and AsyncIO?

A79. • GIL: Global Interpreter Lock — only one thread executes Python bytecode at a time
     • Effect: multi-threading does NOT give CPU parallelism in CPython
     • I/O-bound tasks: threads release GIL during I/O waits — but context switching overhead
     • AsyncIO: single-threaded event loop, cooperative multitasking via coroutines
     • Better for agents: thousands of concurrent API calls with minimal memory/overhead
     • CPU-bound work: use multiprocessing (ProcessPoolExecutor) to bypass GIL entirely

Q80. asyncio.gather() vs asyncio.wait() vs TaskGroup?

A80. • gather(*coros): run all concurrently, return results in input order. Cancels all on first error (if return_exceptions=False)
     • wait(tasks): returns (done, pending) sets. More control: FIRST_COMPLETED, FIRST_EXCEPTION, ALL_COMPLETED
     • TaskGroup (3.11+): structured concurrency — if any task fails, all others are cancelled
     • Use gather: when you need all results and want ordered output
     • Use wait: when you need to process results as they complete
     • Use TaskGroup: when tasks are interdependent and you want clean error handling

Q81. Exponential backoff with jitter?

A81. • import asyncio, random
     • async def retry_with_backoff(func, max_retries=5, base_delay=1):
     •     for attempt in range(max_retries):
     •         try: return await func()
     •         except Exception as e:
     •             if attempt == max_retries - 1: raise
     •             delay = base_delay * (2 ** attempt) + random.uniform(0, 1)  # jitter
     •             await asyncio.sleep(delay)
     • Jitter prevents thundering herd (all clients retrying at the same time)

Q83. __slots__ vs __dict__?

     • Speed: attribute access is slightly faster with __slots__ (direct offset lookup)
     • Use __slots__: when creating millions of instances (e.g., data pipeline records)

Q85. functools.lru_cache limitations for LLM caching?

Q87. Shallow copy vs deep copy?

A87. • Assignment: b = a → both point to SAME object (no copy at all)
     • Shallow copy: b = a.copy() → new outer object, but nested objects still shared
     • Deep copy: b = copy.deepcopy(a) → recursively copies everything, fully independent
     • Data pipelines: modifying a nested dict in a shallow copy mutates the original → bugs
     • Rule: if data has nested structures (list of dicts), always deepcopy before mutation
     • Performance: deepcopy is slower — only use when mutation isolation is needed

Q92. Python garbage collection?

A92. • Reference counting: each object has a refcount; when refcount → 0, immediately freed
     • Problem: circular references (A → B → A) — refcount never reaches 0
     • Cyclic GC: periodically scans for unreachable reference cycles (generational: gen0/1/2)
     • gen0: newly created objects (scanned frequently)
     • gen2: long-lived objects (scanned rarely — expensive)
     • gc.collect() forces a full collection; gc.get_threshold() shows generation thresholds

Q93. Flatten nested JSON?

A93. • def flatten(obj, prefix="", sep="."):
     •     result = {}
     •     for key, val in obj.items():
     •         new_key = f"{prefix}{sep}{key}" if prefix else key
     •         if isinstance(val, dict):
     •             result.update(flatten(val, new_key, sep))
     •         elif isinstance(val, list):
     •             for i, item in enumerate(val):
     •                 if isinstance(item, dict): result.update(flatten(item, f"{new_key}{sep}{i}", sep))
     •                 else: result[f"{new_key}{sep}{i}"] = item
     •         else: result[new_key] = val
     •     return result

Q94. MRO (Method Resolution Order)?

A94. • MRO: order in which Python searches base classes for a method — uses C3 linearization
     • Diamond: D(B, C) where B(A), C(A) → MRO: D → B → C → A (each class appears once)
     • C3 rule: children before parents, left-to-right order preserved, no class repeated
     • Check: MyClass.__mro__ or MyClass.mro()
     • Ensures: super() calls are predictable and don't skip/repeat any class
     • Conflict: if C3 can't linearize (contradictory ordering), Python raises TypeError

Q95. AsyncIO rate limiter?

A95. • class RateLimiter:
     •     def __init__(self, rate): self.semaphore = asyncio.Semaphore(rate)
     •     async def acquire(self):
     •         await self.semaphore.acquire()
     •         asyncio.get_event_loop().call_later(1.0, self.semaphore.release)  # release after 1s
     • Usage: limiter = RateLimiter(10)  # 10 requests/sec
     •   await limiter.acquire(); await make_api_call()
     • More robust: use aiolimiter library (token bucket algorithm)

Q96. ProcessPoolExecutor vs ThreadPoolExecutor?

A96. • ThreadPool: shares memory, limited by GIL → good for I/O-bound (file reads, API calls)
     • ProcessPool: separate processes, bypasses GIL → good for CPU-bound (model inference, numpy)
     • ML pipeline: ThreadPool for data loading/API calls, ProcessPool for feature computation
     • ProcessPool overhead: inter-process communication via pickle (serialization cost)
     • asyncio integration: loop.run_in_executor(pool, blocking_func) for non-async libraries
     • Rule: I/O → threads; CPU → processes; pure async → asyncio (no executor needed)

Q97. Sliding window text chunker?

A97. • def chunk_text(text, chunk_size=500, overlap=50):
     •     words = text.split()
     •     chunks = []
     •     for i in range(0, len(words), chunk_size - overlap):
     •         chunk = " ".join(words[i:i + chunk_size])
     •         if chunk: chunks.append(chunk)
     •     return chunks
     • Overlap ensures context continuity across chunk boundaries
     • For RAG: chunk_size=300-500 tokens, overlap=10-20% of chunk_size

Q98. typing.Protocol vs ABC?

A98. • ABC (Abstract Base Class): requires explicit inheritance — class Foo(MyABC)
     • Protocol: structural subtyping (duck typing) — no inheritance required
     • Protocol: if a class has the right methods, it satisfies the Protocol automatically
     • Preferred when: you can't modify the class hierarchy (third-party code)
     • Example: class Embedder(Protocol): def embed(self, text: str) -> list[float]: ...
     • Any class with an embed(str) → list[float] method satisfies Embedder — no subclassing
